# LLD: Splinter Termination & Provider Handoff Fixes

**Milestone:** M36  
**Status:** draft  
**HLD ref:** §7.3, §10, §15  
**Implements:** `aegis agent exit`, dynamic prompt injection

---

## 1. Purpose

This document addresses two critical failures in the AegisCore orchestration layer:

1.  **Handoff Failure:** The current hardcoded use of `--append-system-prompt-file` in `dispatcher.rs` is Claude-specific. Failover to `gemini-cli` fails because Gemini requires `GEMINI_SYSTEM_MD` environment variable injection.
2.  **Splinter Persistence:** Taskflow splinters (Designer/Implementer) do not terminate because the Watchdog-based detection (legacy `receipt.json` and `[AEGIS:DONE]` strings) has been superseded by M23 Messaging, but the Controller lacks an explicit termination trigger.

---

## 2. Dynamic Provider Prompts

### 2.1 Configuration Changes

`crates/aegis-core/src/config.rs` and `crates/aegis-providers/src/manifest.rs` will be updated to include a `system_prompt_mechanism`.

```rust
#[derive(Debug, Clone, Deserialize, Serialize)]
#[serde(tag = "type", rename_all = "snake_case")]
pub enum SystemPromptMechanism {
    /// Pass via a specific CLI flag (e.g. --append-system-prompt-file)
    Flag { arg: String },
    /// Pass via an environment variable (e.g. GEMINI_SYSTEM_MD)
    Env { var: String },
}
```

### 2.2 Provider Manifest Update

`builtin_providers.yaml` updates:

*   **claude-code:** `system_prompt_mechanism: { type: "flag", arg: "--append-system-prompt-file" }`
*   **gemini-cli:** `system_prompt_mechanism: { type: "env", var: "GEMINI_SYSTEM_MD" }`

### 2.3 Dispatcher Implementation

`Dispatcher::launch_or_insert_plan` and `failover_agent` currently hardcode the flag. They will be refactored to:

1.  Ask the `Provider` for its system prompt injection method.
2.  If `Flag(arg)`, append `arg` and the temp file path to the command arguments.
3.  If `Env(var)`, prepend `export VAR=FILE && ` to the shell script generated by `launch_shell_command`.

## 3. Explicit Agent Termination

### 3.1 `aegis agent exit` Command

A new command is added to the CLI and IPC layer.

*   **Syntax:** `aegis agent exit <AGENT_ID>`
*   **Alternative:** `aegis agent exit self` (resolves ID from `AEGIS_AGENT_ID` env var, which will be injected into all splinter panes).

### 3.2 Termination Workflow (`terminate_splinter`)

The existing `process_receipt` logic in `Dispatcher` is refactored into `terminate_splinter(agent_id)`:

1.  **Transition:** Move agent to `Reporting`.
2.  **M23 Check:** Verify the agent has sent a `Notification` message with `status: "done"` or `"blocked"`. (Optional: could still look for `receipt.json` for backward compatibility, but priority is M23).
3.  **Cleanup:**
    *   Detach and archive Flight Recorder logs.
    *   Kill the tmux pane/window.
    *   Prune the git worktree if `auto_cleanup` is enabled.
4.  **Final State:** Mark agent `Terminated` and archive.

### 3.3 Template Integration

Built-in templates (`taskflow-designer`, `taskflow-implementer`) will be updated to call `aegis agent exit self` as their final shell command.

## 4. Test Strategy

### 4.1 Unit Tests

*   `test_env_prompt_injection`: Verify `dispatcher.rs` correctly generates `export VAR=...` for Gemini.
*   `test_flag_prompt_injection`: Verify it still correctly generates `--flag ...` for Claude.

### 4.2 Integration Tests

*   `test_agent_exit_self`: Spawn a mock agent, run `aegis agent exit <id>`, and assert it is archived and tmux pane is gone.
*   `test_failover_cross_provider`: Trigger failover from a "flag-based" mock provider to an "env-based" mock provider and verify the launch command construction.
